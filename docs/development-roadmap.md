# 翻牌中国象棋开发分阶段规划

## 1. 规划目标

本规划用于把首版产品拆成可连续交付的阶段。每个阶段都应形成清晰的产出物，且尽量以前一阶段的结果作为下一阶段输入，避免过早引入联网、AI 或复杂外围系统。

当前目标固定为：先完成本地双人 MVP，让规则闭环、可玩、可验证。

## 2. 阶段总览

| 阶段 | 名称 | 目标 |
| --- | --- | --- |
| Phase 1 | 规则建模与项目骨架 | 建立棋盘、棋子、局面状态与基础工程结构 |
| Phase 2 | 开局生成与回合状态机 | 完成随机暗置开局、翻牌动作、回合切换 |
| Phase 3 | 走法校验与吃子规则 | 完成明子走法、暗子阻挡、不可直接吃暗子等核心规则 |
| Phase 4 | 将军判定与终局结算 | 完成将军、照将、非法着法拦截与胜负判定 |
| Phase 5 | 本地双人交互与体验补全 | 做出可完整游玩的本地双人 MVP 界面和基础信息反馈 |

## 3. Multica Issue 映射

- Phase 1 对应 `CHE-9`：规则建模与项目骨架
- Phase 2 对应 `CHE-8`：开局生成与回合状态机
- Phase 3 对应 `CHE-10`：走法校验与吃子规则
- Phase 4 对应 `CHE-7`：将军判定与终局结算
- Phase 5 对应 `CHE-11`：本地双人交互与体验补全

## 4. Phase 1: 规则建模与项目骨架

### 目标

- 明确前端技术方案和基础目录结构
- 建立棋盘坐标、棋子实体、阵营、明暗状态的数据模型
- 预留规则引擎与 UI 表现层的边界

### 主要交付物

- 项目初始化完成，可本地运行
- 棋盘与棋子核心数据结构
- 基础局面状态容器
- 开发文档或注释，说明核心模型字段含义

### 完成标准

- 代码中可以稳定表达 9x10 棋盘占用关系
- 每枚棋子具备唯一身份、阵营、棋种、坐标、明暗状态
- 规则计算层与界面层不直接耦合

## 5. Phase 2: 开局生成与回合状态机

### 目标

- 实现“将/帅明置，其他棋子全局随机暗置”的开局生成逻辑
- 实现翻牌算一步的状态流转
- 实现当前行动方、一步完成后的回合切换和合法动作入口

### 主要交付物

- 随机开局生成器
- 翻牌动作处理逻辑
- 回合状态机
- 可用于开发调试的初始局面展示

### 完成标准

- 每次新开局都满足规则限定的格位集合
- 翻开任意暗子后会正确暴露真实棋子信息
- 翻牌不会附带移动或吃子
- 回合在翻牌后正确切换

## 6. Phase 3: 走法校验与吃子规则

### 目标

- 实现所有明子的标准中国象棋走法
- 将暗子纳入阻挡和目标合法性判定
- 实现只能吃对方明子、不能直接吃暗子的规则

### 主要交付物

- 各棋种合法移动与吃子校验
- 路径阻挡判定
- 炮的特殊吃子逻辑
- 非法操作返回明确错误原因

### 完成标准

- 车、马、炮、相/象、士/仕、兵/卒、将/帅的走法全部可验证
- 暗子会阻挡车、炮和将/帅照面的路径
- 任何棋子都不能直接吃未翻开的暗子
- 炮在“有架子但目标为暗子”的情况下会被正确拦截

## 7. Phase 4: 将军判定与终局结算

### 目标

- 实现将军判定
- 拦截让己方将/帅暴露在攻击下的非法着法
- 处理照将和基础终局判定

### 主要交付物

- 将军检测逻辑
- 应将校验逻辑
- 照将检测
- 将死判定和对局结束状态

### 完成标准

- 任意一步执行后都能重新计算双方是否处于将军状态
- 行棋方无法提交让己方将/帅处于非法被将状态的动作
- 将/帅直接照面会被识别为非法局面
- 被将且无合法着法时能正确结算胜负

## 8. Phase 5: 本地双人交互与体验补全

### 目标

- 做出可完整游玩的本地双人界面
- 增加玩家操作反馈，降低规则理解成本
- 提供首版必要的体验补全能力

### 主要交付物

- 可点击翻牌、选子、落子的棋盘界面
- 当前回合提示、将军提示、非法操作提示
- 重新开局能力
- 吃子区或基础对局信息面板

### 完成标准

- 两位玩家能在同一设备上完成一整局游戏
- 非法操作会有明确反馈，不需要依赖控制台排查
- 对局结束后能显示胜负结果并支持重新开局

## 9. 建议实施顺序

1. 先把规则模型和局面状态表达清楚，再谈交互
2. 先把翻牌与回合跑通，再补完整走法和终局
3. 在规则引擎稳定后补 UI 提示、吃子区和重新开局

## 10. 暂不进入本轮计划

以下内容不进入当前阶段拆分：

- 联网对战
- AI 对手
- 匹配系统
- 账号体系
- 复杂复盘与棋谱导出
- 大量视觉特效和高级动画

## 11. v0.5 发布收口状态

### 当前状态

- 3D 对局体验已从“棋盘外壳”升级为真正的 3D 主玩法载体
- 棋子实体化、翻牌/移动/吃子动画、镜头切换等核心 3D 交互已完成并进入主线
- 构建、测试、文档与版本号已同步更新至 v0.5.0
- 发布说明已准确反映当前能力，不再将“仅 3D 棋盘背景”描述为 3D 游戏

### v0.5 已完成范围

- 真正 3D 棋盘、棋子实体与场景交互
- 翻牌、移动、吃子、选中、镜头的 3D 动画反馈
- 性能优化、首屏懒加载与构建产物验证
- 针对 3D 主链路的测试与发布文档

### v0.5 明确不含

- legal-move / highlighted-targets
- 完整动画演出系统
- 复杂相机演出
- 联网、AI、账号等超出本地双人范围的扩展

## 12. v0.4 Beta 发布收口状态

### 当前状态

- 主线已并入 Three.js Phase 1 棋盘外壳，并以 3D Beta 形式继续演进
- 构建阶段已通过懒加载 + vendor chunk 拆分降低首屏阻塞与单 chunk 风险；保留的 `three-core` 独立惰性 chunk 约 724 kB，当前视为可接受的非阻塞项
- 规则层与 `onCellClick(position)` 交互抽象继续保持稳定
- README、玩法说明和版本说明需要与 3D Beta 实际状态保持一致

### v0.4 已完成范围

- 棋盘主视图升级为 3D Beta 外壳
- 保留测试友好的语义棋盘与自动化验证入口
- 补齐发布所需版本号、文档说明与构建验证

### v0.4 明确不含

- legal-move / highlighted-targets
- 完整动画系统
- 复杂相机演出
- 联网、AI、账号等超出本地双人范围的扩展
